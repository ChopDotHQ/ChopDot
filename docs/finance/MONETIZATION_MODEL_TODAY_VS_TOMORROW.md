## FACTS

- The current strongest product thesis is:
  - ChopDot is a shared commitment product
  - not a generic expense app
  - not a booking engine
  - not a custody/yield product
- The current MVP shape is:
  - group commitments
  - contribution state
  - approval / release logic
  - chaptered closeout
  - visible history
  - visible closure
- The current safest launch posture is:
  - coordination first
  - proof second
  - no pooled custody
  - no yield

## INFERENCES

- ChopDot should not monetize like a payments company first.
- ChopDot should monetize like a workflow and trust software company first.
- That means the primary revenue model today should be subscription-led.
- API and infrastructure revenue become more credible later, after the shared commitment kernel is proven.

## ASSUMPTIONS

- ChopDot is still pre-scale and should optimize for clarity and survivability over maximum short-term monetization complexity.
- The product can eventually serve both direct users and other builders, but direct operator value must come first.

## Core Rule

**Today ChopDot sells software that helps groups and organizers move from vague plans to approved, funded, closed action.**

It does **not** sell:
- pooled money management
- yield
- a token
- hidden spread on user funds

## Today: How ChopDot Should Make Money

## 1. Subscription revenue

This should be the primary monetization layer today.

### Who pays

- organizer
- treasurer
- community admin
- club lead
- SMB owner
- provider / operator

### Who should not pay first

- every participant in a casual group

The participant should usually be free or bundled into the organizer account, because charging every participant too early adds friction and weakens adoption.

### What the customer is actually paying for

- visible commitment state
- approval / release control
- closeout clarity
- role-based coordination
- durable history
- reduced organizer pain

### Product framing

You are not charging for “expense splitting.”
You are charging for:

- commitment coordination
- release confidence
- shared closure
- operator control

## 2. Tiered plans

These are the right monetization units for today.

### Organizer Pro

For:
- friend groups
- trips
- retreat organizers
- event leads

What they pay for:
- more commitments
- longer history
- better closeout controls
- better visibility

### Community / Club

For:
- popup communities
- clubs
- collectives
- repeated group organizers

What they pay for:
- multiple active commitments
- member / participant management
- role-aware coordination
- recurring operational use

### SMB / Provider

For:
- small travel operators
- workshop hosts
- service businesses
- event operators

What they pay for:
- deposit and milestone coordination
- release and approval semantics
- cleaner customer-side commitment state
- reduced manual chasing

## 3. Optional service add-ons later

These can exist later, but should not be the core model on day one.

- premium audit / export history
- premium compliance / policy workflows
- premium provider dashboards
- advanced approval rules
- white-label or embedded workflow features

## Today: What ChopDot Should Not Depend On For Revenue

## 1. API revenue

Not as the main model today.

Why:
- the API is not yet the proven product
- builders are not yet the primary buyer
- infrastructure pricing too early can distract from the wedge

API revenue is a **future layer**, not today’s core.

## 2. Transaction take-rate

Not as the primary model today.

Why:
- it pushes ChopDot toward payments-company logic
- it can create legal/compliance complexity
- it risks forcing the product into custody-like behavior

## 3. Yield / treasury spread

Rejected for now.

Why:
- wrong legal posture
- wrong trust posture
- wrong company shape

## 4. Token economics

Rejected for now.

Why:
- not the real value today
- not the moat
- introduces complexity before proof of wedge

## Tomorrow: How ChopDot Can Make Money

Once the commitment kernel is proven, monetization can expand in layers.

## Layer 1: Better subscription revenue

Still the core.

By then the product can charge more for:
- stronger operator workflows
- richer policy controls
- better release logic
- provider workflow tooling
- more durable history and exportability

## Layer 2: Builder / API revenue

This becomes real later.

### What builders would pay for

- commitment APIs
- embedded widgets
- event/history endpoints
- policy and release controls
- builder tools around the shared commitment kernel

### Good pricing unit later

- monthly platform fee
- usage tier
- enterprise support / SLA later

The point is:
- **API is a future monetization layer**
- not the first business model

## Layer 3: Provider / infrastructure revenue

For providers and SMB workflows, ChopDot can later monetize:

- provider dashboard access
- multi-team admin
- workflow automation
- shared commitment operations tooling
- embedded customer commitment journeys

This is stronger than generic “B2B SaaS” because it maps to real operator pain.

## Layer 4: Optional proof / execution fees

This is only acceptable later if:
- the trust value is explicit
- the user understands the action
- the fee is transparent
- ChopDot is not drifting into hidden spread behavior

Good examples later:
- premium verified release workflows
- premium proof / attestation features
- premium execution rails

Bad example:
- quietly taking a percentage of user funds because money passed through the system

## The Monetization Stack

## Today

- primary:
  - subscriptions
- secondary:
  - later premium controls
- deferred:
  - builder/API
  - provider infrastructure
- rejected:
  - yield
  - custody spread
  - token-first economics

## Tomorrow

- primary:
  - stronger subscriptions
- secondary:
  - provider workflows
  - builder/API
  - premium proof / execution features
- still rejected unless the company shape changes deliberately:
  - pooled-funds economics
  - yield
  - token-first business design

## What Protects The Money Model

- strong organizer painkiller
- repeated community / SMB workflows
- durable commitment history
- approval / release semantics generic apps do not handle well
- switching cost once real commitments and histories exist in ChopDot

## What Weakens The Money Model

- if ChopDot looks like a nicer Splitwise
- if users only need it once
- if provider workflows are too weak
- if the product cannot justify premium trust/closure value
- if the company tries to monetize money movement before monetizing workflow value

## The Best Practical Answer

If someone asks “how does ChopDot make money?” the clean answer should be:

**Today ChopDot makes money primarily through subscriptions paid by organizers, communities, and SMB operators who need stronger shared commitment coordination, approval, and closure workflows.**

**Tomorrow ChopDot can add builder/API and provider infrastructure revenue on top of that same commitment kernel.**

## What To Test First

1. Are organizers willing to pay monthly for trusted closeout and release clarity?
2. Are communities willing to pay for recurring commitment coordination?
3. Are SMBs willing to pay for deposit / milestone workflows?
4. Does the product feel premium enough to charge for trust and closure, not just convenience?
5. Which tier has the strongest retention and willingness-to-pay?

## Recommendation

Treat monetization in this order:

1. subscription-led
2. operator-value led
3. provider-workflow led
4. builder/API later
5. no custody/yield/token-first economics unless the company deliberately chooses a different legal and operational posture
