## One-line thesis

ChopDot is building a shared commitment layer that helps groups move from vague plans to approved, funded, closed action, starting with group deposits and reliable closeout.

## What problem ChopDot solves

Groups constantly get stuck between:
- intent
- contribution
- approval
- execution

Today that gap is handled with:
- chats
- spreadsheets
- one organizer fronting the money
- manual chasing
- weak trust semantics

The result is:
- social friction
- operator stress
- unclear readiness
- weak closure

## What ChopDot is

ChopDot is not just an expense app.

It is a shared commitment system built around:
- a canonical commitment object
- participant and role state
- contribution state
- approval / release logic
- chaptered closeout
- visible event history
- explicit closure

## Wedge

The strongest wedge is:
- group deposits before execution

Examples:
- villa deposit
- retreat deposit
- workshop booking deposit
- event funding

This wedge is strong because it has:
- urgency
- real money
- multiple parties
- deadline pressure
- clear success or failure

## Why now

- more groups coordinate financially through chats and fragmented tools
- more popup communities and small operators need lightweight coordination
- tokenization, programmable rails, and agent surfaces are increasing the value of a stable commitment layer
- most existing tools still optimize for tracking or booking, not trusted shared commitment state

## Why ChopDot can matter

ChopDot is aiming to become the control layer for shared commitments.

The long-term expansion path is:
- consumer/group deposit and closeout
- recurring community workflows
- SMB/provider commitment operations
- builder/API and embedded flows
- later agents and selective proof/tokenization adapters

## Moat

The moat is not:
- a wallet integration
- a chain choice
- a generic expense UI

The moat is:
- commitment ontology
- policy/release semantics
- event history
- trust clarity
- expansion from a wedge into a reusable kernel

## Business model

Today:
- subscription-led
- organizers, communities, and SMB operators pay

Tomorrow:
- stronger subscriptions
- premium trust/proof actions
- builder/API pricing
- provider workflow infrastructure

Rejected for now:
- yield
- custody spread
- token-first monetization

## Launch posture

ChopDot should launch as:
- coordination-first
- proof-capable
- non-custodial

This lowers legal and operational risk while preserving long-term expansion optionality.

## Why this can become large

The wedge is narrow, but the primitive is broad.

Many domains share the same underlying need:
- define the shared objective
- coordinate participation
- verify contributions
- gate execution
- preserve closure and history

That is why the same kernel can expand from:
- trips
- events
- communities
- providers
- embedded workflows

## What is already strong

- category framing
- wedge clarity
- architecture direction
- monetization logic
- API-ready posture
- legal/survivability discipline

## What still needs proof

- repeat usage
- willingness-to-pay

## Near-term plan

1. restore the chapter / closeout loop
2. validate it with real organizers and communities
3. prove repeat use and willingness-to-pay
4. then expand into provider and builder surfaces

## What capital would accelerate

- product hardening
- founder validation with real users
- pricing and distribution testing
- stronger operator tooling
- trust/security hardening
- API-ready kernel maturation

## Best summary

ChopDot is becoming a shared commitment layer for groups, communities, and operators. The wedge is group deposits and reliable closeout; the broader ambition is infrastructure for approved, funded, executable shared action.
